# ioctls/ — Per-Subsystem ioctl() Argument Generators

Registry of kernel ioctl-driven subsystems Trinity fuzzes through the `ioctl()` syscall. One file per subsystem (device driver, virtualization interface, filesystem control plane, etc); each declares its ioctl command table and, optionally, hand-rolled argument sanitisers for commands whose args are structs. `syscalls/ioctl.c` picks a group at dispatch time; groups don't call each other.

## Files (59 files, ~14,243 LOC)

Core registry/dispatch (not a per-device group):
- `ioctls.c` (269) — group registration table, fd→group matching (`find_ioctl_group`), the tiered arg-shape picker (`pick_random_ioctl` / `ioctl_arg_for_request`), `dump_ioctls()`. No `REG_IOCTL_GROUP` — this *is* the registry.
- `efault_cache.c` (320) — shm-resident EFAULT-probe cache backing Tier 2 of the arg-shape picker (see below). No `REG_IOCTL_GROUP` either; it's a support module `ioctls.c` calls into.

Largest / structurally distinct per-device files:
- `snd.c` (1887) — ALSA: hwdep, PCM, control, timer, sequencer, mixer sub-ioctl families in one file; 87 `get_writable_struct()` call sites, by far the densest sanitiser file. Multiple independent `sanitise_snd_*` functions dispatched from one top-level switch.
- `vt.c` (968) — virtual terminal / `KD*`/`VT*`/console-font ioctls; 32 `get_writable_struct()` sites, hand-built `termios2` fallback struct when the compat header is missing.
- `drm.c` (690) — DRM core plus vendor extensions (i915, nouveau, radeon, exynos); defines fallback structs/macros for nouveau ioctls not present in all kernel header versions (`DRM_IOCTL_NOUVEAU_GETPARAM` etc, guarded by `#ifndef`).
- `btrfs.c` (537) — btrfs filesystem ioctls (as opposed to `btrfs-control.c`, the tiny 29-line `/dev/btrfs-control` device-manager group).
- `cdrom.c` (503) — CDROM/ATAPI command family, 23 `get_writable_struct()` sites for the various TOC/subchannel/multisession structs.
- `firewire.c` (495) — IEEE1394 character device ioctls, 26 struct-alloc sites.
- `binder.c` (394) — Android binder driver; carries its own copy of `<linux/android/binder.h>`-equivalent definitions (`flat_binder_object`, `binder_write_read`, etc) inline since the header isn't part of the standard kernel-headers package trinity builds against.
- `gpio.c` (362), `vduse.c` (353), `vfs.c` (342), `mtd.c` (338), `autofs.c` (335), `cec.c` (320), `framebuffer.c` (316) — mid-size, each with several `sanitise_*` struct-fillers for their multi-command ioctl surface.

The remaining 44 files (24–282 LOC each) follow the same per-ioctl-table pattern described below, scaled to however many commands the subsystem exposes and whether any need struct-shaped args. Examples: `kvm-vm.c`/`kvm-vcpu.c`/`kvm-system.c` split KVM's ioctl surface by fd type (system fd vs per-VM fd vs per-vCPU fd); `vhost.c`, `nvme.c`, `sg.c`, `i2c-dev.c`, `mei.c`, `usbmon.c`, `pps.c`, `ptp_clock.c`, `rtc.c`, `loop.c`, `nbd.c`, `userfaultfd.c`, `seccomp.c`, `pidfd.c`, `epoll.c`, `dm.c`, `scsi.c`, `hidraw.c`, `input.c`, `joystick.c`, `videodev2.c`, `atm.c`, `socket.c`, `vsock.c`, `vmci.c`, `watchdog.c`, `rfkill.c`, `msr.c`, `hpet.c`, `mce.c`, `dmabuf.c`, `sgx.c`, `uinput.c`, `ext.c`. **`vfio.c` (97) and `iommufd.c` (60) contain only ioctl command tables with no custom sanitisers at all** — every command falls through to the generic `pick_random_ioctl` picker (see Areas of attention).

## Key design decisions

1. **One `struct ioctl_group` per file, self-registered via constructor.** `IOCTL(x)` macro (`include/ioctls.h`) builds `{.request = x, .name = "x"}` table entries; `REG_IOCTL_GROUP(grp)` expands to a `__attribute__((constructor))` function that calls `register_ioctl_group(&grp)` at load time, so files never need to be listed anywhere centrally — dropping a new `.c` file with a `REG_IOCTL_GROUP` line is the entire integration.
2. **Group matching is fd-based, two paths.** `find_ioctl_group()` first tries every registered group's optional `fd_test(fd, &stbuf)` callback (used when devname matching is ambiguous or the fd is anonymous — e.g. `kvm_vm_fd_test()` walks the `OBJ_FD_KVM_VM` object pool rather than checking `/proc/devices`). If no `fd_test` claims the fd, it falls back to matching `devtype` (`DEV_CHAR`/`DEV_BLOCK`/`DEV_MISC`) plus device name against each group's `devs[]` list (resolved via `map_dev()`).
3. **No struct_catalog/ or args/ dependency.** Unlike other struct-producing subsystems in the tree, ioctl arg structs here are filled by hand: a `sanitise_<name>()` function calls `get_writable_struct(sizeof(*s))`/`get_writable_address(size)`, `memset()`s the whole struct to zero, then stamps only the fields the author decided matter (bounded random values, valid-looking enums, etc), leaving the rest zeroed rather than catalog-random. A static-analysis tripwire enforces the "memset before field stores" half of this pattern (`scripts/check-static/ioctl-struct-memset.sh`) because `get_writable_struct()` returns recycled pool memory that can leak previous-syscall residue back to the kernel (or be read as attacker-influenced-but-unintended field values) if left unzeroed.
4. **Three-tier arg-shape picker in `ioctls.c` (`ioctl_arg_for_request`)**, run before any per-group custom sanitiser, then overridden per-command where a custom sanitiser exists:
   - Tier 1 (decoded): if the request was built with `_IO*()` and carries a non-`_IOC_NONE` direction, trust `_IOC_SIZE()`/`_IOC_DIR()` for buffer size and read/write direction; seed with random bytes for `_IOC_WRITE`.
   - Tier 2 (EFAULT-probe cache, `efault_cache.c`): for legacy raw-constant ioctls (TCGETS-style, sockios) with no shape info in the request number, call the ioctl once with a bogus pointer arg and classify by errno (`EFAULT`→pointer, `EINVAL/ENOTTY/ENOSYS/0`→scalar, `EBADF/ENXIO`→unknown/mismatched fd). Verdict is cached fleet-wide in shm, keyed by `(group_idx, request)`, since the kernel's dispatch table is process-independent. Probes twice with different bogus values and only commits if both agree.
   - Tier 3 (legacy 50/50): coin-flip between a random scalar and a writable page, used when Tier 2 is inconclusive or the group opted out of probing.
5. **EFAULT-probe opt-out list** (`efault_cache.c`, `efault_optout_devs[]`/`efault_optout_names[]`): groups whose ioctls allocate real kernel state on the dispatch path *before* arg validation — `kvm`, `vhost-{net,vsock,scsi,vdpa}`, `iommu`, `vfio`, `loop-control` (by devname) and `kvm_vcpu` (by group name, since vCPU fds have no `/proc/devices` entry and some vCPU ioctls like `KVM_NMI`/`KVM_SMI` are side-effecting `_IO()` calls that would fire for real during a probe) — skip Tier 2 entirely to avoid leaking vcpus/rings/containers/loop devices on every probe.
6. **Post-sanitise scrub against shared regions.** `pick_random_ioctl()` and `syscalls/ioctl.c`'s `sanitise_ioctl()` both call `avoid_shared_buffer_out()` on `a3`/`a4`/`a5`/`a6` — custom sanitisers that overwrite `a3` with their own struct pointer are covered by the second scrub in `syscalls/ioctl.c`, run unconditionally after the group's `.sanitise()` returns, so a struct pointer accidentally aliasing trinity's shm bookkeeping (childdata/stats/kcov buffers) can't be handed to the kernel for `copy_to_user`.

## Integration points

- `syscalls/ioctl.c` — the actual `ioctl()` syscall entry (`sanitise_ioctl()`): picks a group via `find_ioctl_group(rec->a1)` (99% of the time) or `get_random_ioctl_group()` (1-in-100, deliberately mismatching fd and group to explore misuse), calls `grp->sanitise(grp, rec)`, then re-scrubs `a3`.
- `include/ioctls.h` — shared types: `struct ioctl`, `struct ioctl_group`, `IOCTL()`/`REG_IOCTL_GROUP()` macros, `register_ioctl_group`/`find_ioctl_group`/`get_random_ioctl_group`/`pick_random_ioctl`/`dump_ioctls` prototypes.
- `fds/kvm.c` (and analogous per-subsystem fd modules) — populate the `OBJ_FD_*` object pools that `fd_test` callbacks (e.g. `kvm_vm_fd_test`) walk to recognize programmatically-created fds with no `/proc/devices` presence.
- `shm.c`/`shm` — backs both the EFAULT-probe cache (`shm->ioctl_efault_cache[]`, lock-free CAS insert) and per-group dispatch counters some groups bump (e.g. `kvm-vm.c` increments `shm->stats.kvm_vm_ioctls_dispatched`).
- `random.c`/`rnd.c` — `get_writable_address()`/`get_writable_struct()` (pool-backed buffer allocation), `rnd_modulo_u32()`/`rnd_u64()`/`RAND_BOOL()`/`ONE_IN()` for value generation.
- `scripts/check-static/ioctl-struct-memset.sh` — CI-time regex tripwire enforcing the memset-before-field-store convention across every `sanitise_*` function in this directory.
- `syscalls/io_uring_register.c` — references `pick_random_ioctl()`'s arg-scrub ordering in a comment, implying a parallel (not shared) arg-shape convention for `io_uring_register`'s own opcode table.

## Areas of attention

1. **`vfio.c` and `iommufd.c` have zero custom sanitisers** — both are `.sanitise = pick_random_ioctl` with no per-command struct fillers, despite being complex, security-sensitive kernel surfaces (IOMMU page-table/DMA-mapping management, PCI device passthrough) with several commands that take multi-field structs (`vfio_iommu_type1_dma_map`, `iommu_ioas_map`, etc). Every dispatch to these gets whatever Tier 1/2/3 generic buffer-of-random-bytes the picker produces, which likely fails validation before reaching interesting kernel logic — the opposite of `kvm-vm.c`'s approach for a comparably sensitive subsystem.
2. **`kvm-vm.c` documents its own struct-shaped-arg rationale in a comment**: Tier 1's `_IOC_SIZE`-based generic buffer under-sizes `kvm_irq_routing`'s flexible array tail and leaves fixed structs' fields (slot indices, alignment, `guest_phys_addr` bounds) unconstrained, so almost every dispatch would bounce on `-EINVAL` without hand-written sanitisers — the same argument likely applies unaddressed to `vfio.c`/`iommufd.c`.
3. **`snd.c` at 1887 LOC is a monolith** spanning hwdep, PCM, control, timer, sequencer and mixer ioctl families with 87 separate struct-allocation sites — the highest sanitiser density in the directory and the natural first split-candidate if this directory follows the pattern of other Trinity subsystems that got broken up (see the `kvm-vm.c`/`kvm-vcpu.c`/`kvm-system.c` split noted in `kvm-vm.c`'s own header comment, and the `ioctls/vt.c` vs `ioctls/btrfs.c`/`btrfs-control.c` precedent for splitting a driver's ioctl surface by fd type or subsystem).
4. **EFAULT-probe side effects are opt-out by string list, not derived from the kernel.** `efault_optout_devs[]`/`efault_optout_names[]` in `efault_cache.c` is a manually maintained list; a new group whose dispatch path allocates kernel state before arg validation (a new vhost transport, a new virtualization ioctl set) silently leaks that resource on every EFAULT probe until someone notices and adds it to the list.
5. **No shared struct-generation infrastructure.** Every `sanitise_*` function hand-codes field values (bounded `rnd_modulo_u32()` calls, `RAND_BOOL()` flag picks); there is no catalog-driven or reflection-based struct filler for this directory (`grep -rn struct_catalog ioctls/` — zero hits), so field coverage and bounds-plausibility depend entirely on the judgment of whoever wrote each `sanitise_*` function, and are not automatically kept in sync with kernel struct changes.

## Summary

`ioctls.c` is a self-registering table of `struct ioctl_group`s, one per kernel subsystem file in this directory; `syscalls/ioctl.c` matches an fd to a group and lets the group's `.sanitise` callback pick a command and shape its argument. Argument generation is a three-tier fallback (decoded `_IOC_SIZE`/`_IOC_DIR` → EFAULT-probe cache → random scalar/page) for commands with no custom handler, overridden per-command by hand-written `sanitise_*` functions (memset-then-stamp-fields) where a subsystem author judged the generic tiers insufficient — a judgment applied unevenly across the directory (KVM's VM/vCPU ioctls are heavily hand-tuned; VFIO and IOMMUFD, comparably complex and security-relevant, are not).
