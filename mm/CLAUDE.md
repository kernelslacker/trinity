# mm/ — Memory-Management Fuzzing-Target Subsystem

Owns the fuzzed process's synthetic VMA pools (anon/file/testfile mmap objects), deliberate page-fault injection (read and write), mlock/memlock accounting, initial-mapping and image-segment bookkeeping, hugepage size-encoding helpers, and a VMA-count watchdog. This is the layer that hands out addresses/lengths for ARG_ADDRESS/ARG_MMAP args and then actively touches the memory behind them to trigger kernel fault paths.

## Key design decisions

1. **Two-tier pool model (OBJ_GLOBAL vs OBJ_LOCAL)** — `setup_initial_mappings()` builds one shared/global ANON snapshot pre-fork; `init_child_mappings()` clones it (by-pointer, COW-backed) into each child's OBJ_LOCAL pool. FILE/TESTFILE pools are lazily cloned the same way (`clone_global_mmap_pool`). Runtime mmap()/munmap() by a child only ever mutates its own OBJ_LOCAL pool.
2. **Pool-type masking (maps.c `pick_mmap_pool_type`)** — a per-child `mmap_pool_nonempty_mask` restricts the uniform 1-of-3 draw to non-empty OBJ_LOCAL pools, avoiding guaranteed-miss draws once FILE/TESTFILE drain; OBJ_GLOBAL (parent-side) keeps the plain uniform pick since it has no occupancy shadow.
3. **Layered corruption defense on every pool pick** (`get_map_handle`) — three independent guards run before any deref: user-VA-band check on the obj pointer (`obj_ptr_in_user_va_band`), pool-slot validity via `objpool_check` (`obj_pool_slot_check`; validates obj->obj_type against the pool the draw landed on -- same gate the sibling `addr_in_local_runtime_map` / `invalidate_mmap_pool_range` walks use), and a size sanity cap at GB(4) (`map_size_in_range`). Each has a dedicated per-type stats counter; failing any retries up to 1000 times, then triggers `maybe_refill_local_anon_pool`.
4. **Destructor split by scope** — `map_destructor` (OBJ_LOCAL, libc strdup/malloc names, plain `free`) vs `map_destructor_shared` (OBJ_GLOBAL, shared-heap names via `alloc_shared_str`/`free_shared_str`). Both re-validate `ptr`/`extent` against the VA band and GB(4) cap before `munmap`/`untrack_shared_region`, since a scribbled map struct could otherwise drive a wild unmap of trinity's own bookkeeping.
5. **tracked_size vs size split** — `map->size` is the consumer-walkable, fstat-clamped extent (never past real file EOF); `map->tracked_size` is the actual VMA extent the kernel mapped, used for `track_shared_region`/`untrack_shared_region` and `munmap` so a clamp-for-safety on `size` doesn't leak the tail of the VMA in the shared-region tracker.
6. **Live TOCTOU handling in the fault injectors** — both `random_map_readfn`/`random_map_writefn` snapshot the map, re-`fstat` file-backed extents to clamp against a shrinking file, then install a scoped SIGSEGV/SIGBUS handler (`sigsetjmp`/`siglongjmp`) that only intercepts faults whose `si_addr` falls inside the active walk range; anything else (or a spoofed/self-sent signal) is passed through to the normal `child_fault_handler` crash path. Handler state is per-file `volatile` statics, matching the same pattern used in `childops/madvise-pattern-cycler.c`.
7. **Per-call fault-walk cap (32 pages)** — both fault-read and fault-write cap the pages touched per call to bound the mprotect-driven TLB-shootdown IPI storm on large mappings; offset/index selection stays uniform over the full mapping, only the iteration count is capped.
8. **mlock budget accounting is `__thread`, not per-process** — tracks cumulative locked bytes against `RLIMIT_MEMLOCK` so repeated mlock draws don't uniformly EAGAIN once the cap is exhausted; deliberately returns `page_size` rather than 0 when over budget so the call still exercises the kernel's rlimit-check path instead of no-op'ing on `len=0`.
9. **VMA-pressure watchdog is detect-and-back-off only (v1)** — hysteresis (80% on / 70% off) against `/proc/sys/vm/max_map_count`, disabled entirely above 250k max VMAs (read cost becomes prohibitive and kernel headroom makes the concern moot). No active recovery (VMA recompose) is attempted; design note in the file explains why (ownership ambiguity across pool types, coverage-cost tradeoff, added failure-mode risk).
10. **Image-segment registration is a startup one-shot** — `dl_iterate_phdr()` runs once before `fork_children()`; only PT_LOAD segments with `PF_W` are tracked (read-only text/rodata mprotect strips just SIGSEGV on next fetch, already handled by the fault machinery). No re-scan on hypothetical future `dlopen()`.
11. **Hugepage shift probing is lazy and cached** — `/sys/kernel/mm/hugepages/` is scanned once per process; falls back to a fixed architecture-agnostic shift list (2MB/32MB/256MB/1GB/8GB) if the sysfs directory is absent, deliberately allowing -EINVAL as a fuzzable outcome.

## Areas of attention

1. **maps.c is doing three jobs at 1418 LOC** — pool pick/validate/refill, destructor pairs (local vs shared), and arg-plumbing (`common_set_mmap_ptr_len`, type resolution walk) are all in one file. The corruption-defense layer (obj-ptr band check, alloc-track lookup, size cap) is duplicated inline across `get_map_handle`, `map_destructor`, and `map_destructor_shared` rather than factored into one shared validator.
2. **`common_set_mmap_ptr_len`'s type-resolution walk is O(n) over all three pools** on every call that needs `out_type` (used by `post_munmap`'s WHOLE-destroy path) — a linear scan comparing `obj` pointers, with its own scan-length stats counters acknowledging the cost.
3. **Fault-injection signal handlers are duplicated near-verbatim** between fault-read.c and fault-write.c (armed flag, lo/hi range, sigsetjmp/siglongjmp dance) — the file-top comments in both explicitly note the same pattern exists a third time in `childops/madvise-pattern-cycler.c` and `childops/pagecache-canary-check.c`. `-Wclobbered`/volatile requirements make hoisting nontrivial but four independent copies is a real drift risk.
4. **`/proc/self/maps` parsing appears in three places** (`maps.c::proc_maps_check`, `vma-pressure.c::vma_pressure_count_self_vmas`, `stats.c::count_proc_maps_lines`) with different failure semantics (true-on-error vs ULONG_MAX-on-error) — a shared parser deliberately was not built to avoid include-surface entanglement, per vma-pressure.c's note, but this means fixing a `/proc/self/maps` line-format edge case requires touching three files.
5. **VMA-pressure watchdog has no active recovery path** — a child that trips the high-water latch can only wait for natural attrition or its own exit/respawn; under a host with fast VMA growth and slow child turnover this is a documented (not hidden) coverage gap.

## Summary

mm/ supplies both the raw material (VMA pools seeded pre-fork, cloned per-child, refilled lazily) and the active fault-injection machinery (read/write walkers with TOCTOU-safe signal recovery) that the rest of trinity uses to exercise page-fault, mmap/mprotect/mlock, and hugepage kernel paths. Everything here is defensive by construction: pool consumers assume a fuzzed syscall may have scribbled any pointer/size field, and every pick/destroy path re-validates before dereferencing. The two watchdog-style modules (vma-pressure.c, image-segments.c) exist purely to keep trinity's own process alive and its own memory unmolested while the fuzzed syscalls run.
